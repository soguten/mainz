title: I18n
summary: Use dictionaries for UI copy, and use locale-aware data loading for domain content that already lives in external sources.
---

## Two different localization jobs

In Mainz, locale-aware routing and translation runtime are related, but they do
not solve the same problem.

The framework supports two common localization paths:

- `dictionaries` for predictable UI copy
- locale-aware external loading for domain content

Both can live in the same app.

## Use dictionaries for UI copy

Use `dictionaries` when the app already knows which text goes where.

Typical examples:

- navigation labels
- buttons
- validation messages
- short marketing copy
- diagnostics text

Declare the dictionaries in `defineApp({ i18n })`:

```tsx title="app.ts"
import { defineApp } from "mainz";
import en from "./i18n/locales/en.ts";
import pt from "./i18n/locales/pt.ts";

export const app = defineApp({
  id: "site",
  i18n: {
    locales: ["en", "pt"],
    defaultLocale: "en",
    localePrefix: "except-default",
    dictionaries: {
      en,
      pt,
    },
  },
  pages: [HomePage],
});
```

Then consume the framework-owned runtime from `mainz/i18n`:

```tsx title="HeroIntro.tsx"
import { t } from "mainz/i18n";

export function HeroIntro() {
  return (
    <section>
      <h1>{t("hero.title")}</h1>
      <p>{t("hero.lead")}</p>
    </section>
  );
}
```

The runtime also owns locale state and formatting helpers:

```ts
import { getLocale, setLocale, t } from "mainz/i18n";

getLocale();
setLocale("pt");
t("greeting", { name: "Alex" });
```

## Use external loading for domain content

Not every localized text should become a dictionary key.

When the content already belongs to a CMS, database, markdown collection, or
another external source, use the resolved route locale in `entries()` or
`load()` and fetch the localized content directly.

Typical examples:

- documentation articles
- blog posts
- product descriptions
- CMS pages
- localized database records

Example shape:

```tsx title="Docs.page.tsx"
import { Locales, Page, Route } from "mainz";

@Route("/docs/:slug")
@Locales("en", "pt")
export class DocsPage extends Page {
  static override async entries({ locale }) {
    return await loadDocEntriesFromCms(locale);
  }

  override async load({ params, locale }) {
    return await loadDocFromCms({
      slug: params.slug,
      locale,
    });
  }
}
```

In that model:

- route locale still comes from `defineApp({ i18n })`
- UI copy can still come from `dictionaries`
- article or CMS content comes from the external source in the correct locale

Mainz does not require every localized text in the product to be represented as
a dictionary entry.

## Routing still owns locale resolution

`defineApp({ i18n })` is still the source of truth for locale-aware routing:

- `locales`
- `defaultLocale`
- `localePrefix`

`@Locales(...)` remains the page-level way to restrict participation inside that
app locale set.

So the split is:

- routing decides which locale the route is in
- `dictionaries` translate predictable UI copy
- `load()` and `entries()` may fetch external localized content directly

## Fallbacks are not route changes

If you add translation fallback behavior, that fallback should affect
translation lookup only.

It should not:

- change the resolved route locale
- change canonical URLs
- change alternate links
- change route matching

For the fallback boundary, follow the same model described by the internal RFCs:

- route locale is one concern
- translation/content fallback is another concern
